# 광고 · 업데이트 셋업 & 사용 가이드

RecycleAI에 적용된 **광고(AdMob)** 와 **인앱 업데이트(Google Play)** 를 처음부터 설정하고 운영하는 방법을 정리한 문서입니다. 바이브코딩 입문자도 따라 할 수 있도록 단계별로 적었습니다.

> 운영 중 빠른 참조(켜고 끄기, Remote Config 키 등)는 [`admin_operations_guide.md`](admin_operations_guide.md) 를 함께 보세요. 이 문서는 "처음 셋업 + 실제 배포" 흐름에 집중합니다.

---

## 0. 한눈에 보기

| 항목 | 무엇 | 어디서 설정 | 코드 위치 |
|------|------|-------------|-----------|
| 배너 광고 | 결과 카드에 띄우는 작은 광고 | AdMob 콘솔 | `ads/AdIds.kt`, `ui/CardSheet.kt` |
| 보상형 광고 | 횟수 초과 시 보고 충전하는 영상 광고 | AdMob 콘솔 | `ads/AdIds.kt`, `ads/RewardedAdManager.kt` |
| 횟수 제한 | 하루 무료 분석 횟수 | Firebase Remote Config | `data/RemoteConfigManager.kt`, `data/ScanLimitManager.kt` |
| 인앱 업데이트 | 앱 안에서 새 버전 안내·설치 | Google Play Console | `update/InAppUpdateManager.kt`, `MainActivity.kt` |

현재 등록된 실제 값:

```text

배너 광고 단위 ID              : ca-app-pub-9847865524181124/6430260715
보상형 광고 단위 ID            : ca-app-pub-9847865524181124/9442447187
```

## 1. AdMob 광고 셋업

### 1-1. AdMob 앱 등록 (최초 1회)

1. [apps.admob.com](https://apps.admob.com) 로그인 → **앱(Apps) → 앱 추가**
2. 플랫폼 **Android** 선택, 아직 스토어 출시 전이면 "아니요(앱이 아직 등록 안 됨)" 선택
3. 생성되면 `ca-app-pub-...~...` 형태의 **앱 ID**가 나옵니다.
   - 이 ID는 이미 `AndroidManifest.xml`의 `APPLICATION_ID` 메타데이터에 들어가 있습니다.

### 1-2. 광고 단위 생성 (배너 + 보상형, 각각 1개)

광고 유형마다 따로 만듭니다. **앱 → 광고 단위 → 광고 단위 추가**

- **배너**: 형식 `배너` 선택 → 이름 예) `결과카드 배너` → 생성
- **보상형**: 형식 `보상형` 선택 → 이름 예) `횟수충전 보상형` → 생성
  - 리워드 수량/상품(1, "Reward")은 **앱이 사용하지 않으므로 기본값 그대로** 둬도 됩니다. (충전 횟수는 앱의 `daily_scan_limit`이 결정)
  - 리워드 설정은 생성 후에도 수정 가능합니다.
- 공통: **파트너 입찰** 체크박스는 **체크하지 않습니다.** (AdMob만 사용)

생성하면 각각 `ca-app-pub-.../숫자` 형태의 **광고 단위 ID**가 나옵니다.

### 1-3. 코드에 실제 ID 넣기

모든 광고 ID는 한 파일에서 관리합니다: `app/src/main/java/app/trashai/ads/AdIds.kt`

```kotlin
// 실제 ID (release 빌드에서 사용)
private const val PROD_BANNER = "ca-app-pub-9847865524181124/6430260715"
private const val PROD_REWARDED = "ca-app-pub-9847865524181124/9442447187"
```

- **debug 빌드**: 자동으로 Google **테스트 광고**가 나옵니다. → 개발 중 실수로 내 광고를 클릭해 계정이 정지되는 사고를 막아줍니다.
- **release 빌드**: 위 실제 ID가 나갑니다.
- 이 분기는 `BuildConfig.DEBUG`로 자동 처리되므로 따로 건드릴 필요 없습니다.

### 1-4. 배너 광고 켜고 끄기

`ui/CardSheet.kt` 상단 플래그로 결과 카드 배너 표시를 토글합니다.

```kotlin
private const val SHOW_AD_BANNER = true   // false 로 두면 배너 숨김
```

### 1-5. 보상형 광고 동작 흐름

- 무료 분석 횟수를 다 쓰면 충전 안내 카드가 뜹니다.
- "광고 보고 N회 충전하기" → `RewardedAdManager`가 광고를 표시.
- **영상을 끝까지 본 경우에만** 충전되고 카메라 화면으로 복귀합니다.
- 광고가 아직 안 불러와졌으면 로딩 후 자동 표시, 실패 시 "잠시 후 다시" 안내(충전 안 됨).

### 1-6. 전면 팝업 광고를 추가하고 싶다면 (선택)

지금은 배너·보상형만 있습니다. 앱 실행 시 전면 광고가 필요하면 **전면 광고(App Open / Interstitial)** 광고 단위를 AdMob에서 따로 만들고, 별도 매니저를 추가 구현하면 됩니다. (현재 미구현 — 필요 시 요청)

## 2. AI 스캔 횟수 제한 (Firebase Remote Config)

서버에서 원격으로 켜고 끄거나 횟수를 바꿀 수 있습니다. 앱 재배포 불필요.

| 키 | 타입 | 기본값 | 설명 |
|---|---|---:|---|
| `limit_enabled` | Boolean | `false` | 일일 횟수 제한 사용 여부 |
| `daily_scan_limit` | Number | `5` | 하루 무료 분석 횟수 |

설정: Firebase Console → Remote Config → 위 키 값 변경 후 **게시(Publish)**. 앱은 실행/복귀 시 최신 값을 가져옵니다.

자세한 내용은 [`admin_operations_guide.md`](admin_operations_guide.md) 참고.

## 3. 인앱 업데이트 셋업

앱 안에서 새 버전을 감지해 "업데이트 → 다시 시작"을 안내합니다. (Google Play In-App Updates, Flexible 방식)

코드 위치: `update/InAppUpdateManager.kt`, `MainActivity.kt`

### 3-1. 핵심 개념 (꼭 이해하기)

- **소스 코드는 항상 로컬(내 PC)에 있습니다.** Play에 올리는 건 소스가 아니라 **빌드 결과물(AAB) 파일 하나**뿐입니다.
- 인앱 업데이트는 폰의 **Play 스토어 앱에게** "더 새 버전 있어?"라고 묻는 기능입니다.
  - **Android Studio로 직접 설치(USB/APK 사이드로딩)** 한 앱 → Play가 모름 → 안내 안 뜸 (정상)
  - **Play 스토어로 설치**한 앱 → Play가 버전 비교 → 새 버전 있으면 안내 뜸
- 즉, 업데이트 안내 기능은 **Play로 설치된 앱에서만** 테스트됩니다. 나머지 기능(카메라·광고·DB)은 지금처럼 로컬 설치로 다 확인됩니다.

### 3-2. APK vs AAB

| | APK (`.apk`) | AAB (`.aab`) |
|---|---|---|
| 용도 | 폰에 바로 설치 | Play에 업로드하는 원본 묶음 |
| 폰에 바로 설치 | 됨 | 안 됨 |
| Play 출시 | 불가(2021.8~) | **필수** |

Play 스토어 출시에는 반드시 **AAB**가 필요합니다.

### 3-3. AAB 만드는 법 (Android Studio)

1. 메뉴 **`Build` → `Generate Signed Bundle / APK...`**
2. **`Android App Bundle`** 선택 → Next
3. 서명 키(keystore) 선택, 처음이면 `Create new...`로 생성
   - ⚠️ **keystore 파일과 비밀번호는 절대 분실 금지.** 잃어버리면 같은 앱으로 업데이트를 못 올립니다.
4. Build variant **`release`** → Finish
5. 결과 파일: `app/release/app-release.aab`

### 3-4. 새 버전 내보내기 (업데이트 배포)

1. `app/build.gradle.kts`에서 **`versionCode`를 반드시 올림** (예: `1` → `2`), `versionName`도 갱신
   ```kotlin
   versionCode = 2          // 이전보다 큰 숫자 (필수)
   versionName = "0.2.0"
   ```
2. AAB 빌드 (3-3)
3. Play Console에 업로드 → 출시
4. 사용자 앱이 다음 실행 때 "업데이트 준비 완료 → 지금 다시 시작" 안내를 띄움

> versionCode를 안 올리면 Play가 "같은 버전"으로 보고 업데이트로 인식하지 않습니다. **버전 올리기를 잊지 마세요.**

### 3-5. 인앱 업데이트 테스트 방법

1. Play Console에서 앱 생성 후 **내부 테스트(Internal testing)** 트랙에 AAB(versionCode 1) 업로드
2. 테스트 링크로 폰에 설치 (→ "Play가 설치한 앱"이 됨)
3. versionCode 2로 올린 AAB를 같은 트랙에 업로드
4. 폰에서 앱 실행 → 업데이트 안내가 뜨는지 확인

### 3-6. 강제(즉시) 업데이트로 바꾸려면

필수 업데이트가 필요하면 `InAppUpdateManager`의 `AppUpdateType.FLEXIBLE`을 `AppUpdateType.IMMEDIATE`로 변경합니다. 전체 화면 차단형이 되어 업데이트해야만 앱을 계속 쓸 수 있습니다.

## 4. 실제 배포 전 체크리스트

- [ ] `AdIds.kt`의 `PROD_BANNER`, `PROD_REWARDED`가 실제 발급 ID인지 확인
- [ ] release 빌드에서 테스트 광고가 안 나가는지(= `BuildConfig.DEBUG` 분기 유지) 확인
- [ ] `SHOW_AD_BANNER` 값이 의도대로인지 확인
- [ ] `limit_enabled` / `daily_scan_limit` 기본값이 출시 정책과 맞는지 확인
- [ ] 보상형 시나리오 테스트: 미로드 / 로딩 후 자동표시 / 중도이탈(충전X) / 완료(충전O)
- [ ] `versionCode`를 이전보다 올렸는지 확인
- [ ] keystore 파일·비밀번호 안전하게 백업했는지 확인
- [ ] 개인정보 처리방침에 광고 SDK·카메라·위치 권한 목적 반영
- [ ] 새 광고 단위는 게재까지 최대 1시간 걸릴 수 있음(초기 빈 광고는 정상)

## 5. 자주 막히는 부분 (FAQ)

**Q. 소스가 다 로컬에 있는데 "Play로 설치"가 무슨 말?**
→ Play에는 소스가 아니라 빌드 결과물(AAB)만 올립니다. 폰에 넣는 경로만 "Play 스토어 경유"로 바뀌는 것이고, 인앱 업데이트 안내는 그때만 동작합니다.

**Q. 광고가 안 떠요.**
→ debug 빌드는 테스트 광고만 나옵니다. 새로 만든 실제 광고 단위는 게재까지 최대 1시간 걸릴 수 있습니다. `SHOW_AD_BANNER`가 `true`인지도 확인하세요.

**Q. 업데이트 안내가 안 떠요.**
→ Android Studio로 직접 설치한 앱에서는 원래 안 뜹니다(정상). Play(내부 테스트) 설치본 + 더 높은 versionCode가 있어야 뜹니다.
