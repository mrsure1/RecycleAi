# AI 바이브 코딩(Vibe Coding)으로 완성하는 실전 모바일 앱 개발 가이드
> **프로젝트 사례**: RecycleAI (스마트 분리배출 가이드 모바일 앱)  
> **대상 독자**: AI 및 실전 앱 개발을 배우고 있는 학원생 및 주니어 개발자  
> **문서 목적**: 바이브 코딩의 개념부터 개발 툴, AI 모델, DB, 공공 API, UI/UX 설계까지 전 과정을 설명하고 향후 PPT 발표 자료로 활용

## 1. 바이브 코딩(Vibe Coding)의 시작: 기획과 프롬프트

### 1.1 바이브 코딩이란?
과거처럼 개발자가 문법을 외워 코드 한 줄 한 줄을 직접 타이핑하는 대신, **AI 에이전트에게 제품의 비전, 페르소나, 아키텍처의 ‘느낌(Vibe)’을 대화하듯 설명하고 조율하며 시스템을 구축해 나가는 현대적인 개발 패러다임**입니다.

### 1.2 시작 단계에서 가장 먼저 해야 할 일
바이브 코딩을 할 때 무작정 “재활용 앱 만들어줘”라고 하면 실패합니다. AI가 명확한 방향을 잡을 수 있도록 다음 3가지 핵심 문서를 먼저 작성하고 공유해야 합니다.
1. **`problem.md` (문제 정의 및 페르소나)**: “고양시 일산으로 이사 온 60대 박정호 씨가 지역별로 다른 분리수거 룰 때문에 혼란과 수치심을 겪는다”처럼 구체적인 타깃과 고통(Pain Point)을 명시합니다.
2. **`vision.md` (핵심 가치 제안)**: 텍스트 검색의 피로감을 없애고, 카메라 스캔과 핀치 줌 바텀시트로 직관적인 확신을 준다는 제품의 비전을 정의합니다.
3. **`PRD.md` (제품 요구사항 정의서)**: 헤더, 카메라 뷰포트, 바텀시트 등 화면별 필수 기능과 우선순위(P0, P1)를 명확히 나눕니다.

## 2. 개발 툴 및 AI 모델 스택

```
[ 개발 환경: Antigravity IDE ] ──(프롬프트/컨텍스트)──> [ 메인 브레인: Gemini 3 Flash ]
         │                                                      │
         ├── 파일 시스템 및 터미널 자동 제어                         ├── 실시간 코드 생성 및 리팩토링
         └── MCP 서버 (Firebase, Codex, NotebookLM 등)           └── 복잡한 에러 및 의존성 디버깅
```

### 2.1 개발 툴: Google DeepMind Antigravity
* **개념**: 터미널, 파일 시스템, 브라우저 서브에이전트, MCP(Model Context Protocol) 서버를 통합 제어하는 강력한 에이전틱 IDE 환경입니다.
* **학원생들을 위한 설명 포인트**: 개발자가 코드를 수정해 달라고 요청하면 Antigravity 에이전트가 직접 프로젝트 폴더를 탐색(`list_dir`, `grep_search`)하고, 파일을 수정(`replace_file_content`)하며, 필요 시 터미널 명령어를 실행하여 빌드와 테스트까지 수행하는 자율형 페어 프로그래밍 도구입니다.

### 2.2 메인 AI 모델: Gemini 3 Flash
* **개념**: 구글의 최신 경량·초고속 멀티모달 AI 모델입니다.
* **선택 이유 (Why Gemini 3 Flash?)**: 
  1. **압도적인 속도와 가성비**: 수만 줄의 코드 콘텍스트를 즉각적으로 읽고 리팩토링이나 디버깅 방향을 제시합니다.
  2. **멀티모달 비전 추론**: UI 캡처 화면이나 복잡한 아키텍처 다이어그램을 정확히 이해하고 코드로 변환하는 능력이 탁월합니다.

## 3. 데이터베이스 (DB) 아키텍처: 로컬과 클라우드의 조화

```
[ 카메라 / 드래그 스캔 ]
         │
         ├── 1차: 로컬 DB (자체 730개 핵심 품목 SQLite / Room) ──> 0.1초 내 초고속 매칭 (배터리/발열 보호)
         │
         └── 2차: Gemini (이미지→키워드) ──> SQLite Grounding
         
[ 빌드 타임 ] wasteguide 크롤 + MOIS import → assets SQLite (조례·배출일정·E-순환)
```

### 3.1 1차 로컬 DB (자체 730개 핵심 품목)
* **구현 방식**: 안드로이드 내부 저장소(SQLite / Room)에 730개의 핵심 재활용 품목 메타데이터를 캐싱합니다.
* **학원생들을 위한 설명 포인트**: “왜 모든 걸 클라우드 AI로 처리하지 않았을까요? 모바일 기기에서 매번 무거운 AI 모델이나 서버 통신을 거치면 폰이 뜨거워지고 배터리가 금방 닳습니다. 일상 쓰레기의 85%는 기기 내부에서 0.1초 만에 매칭되도록 설계한 것이 핵심 기술 결정입니다.”

### 3.2 지역 조례·배출 일정 (번들 SQLite)
* **구현 방식**: `app_region_ordinance`(wasteguide), `app_mois_disposal`(행안부 MOIS)를 앱 assets에 포함. 5↔7자리 지역코드는 `build_region_mois_map.py`로 import 전에만 매핑.
* **발표 포인트**: “배출 요일을 쓸 때마다 서버를 두드리지 않습니다. 앱은 오프라인 DB만 읽고, 데이터 갱신은 출시 전 파이프라인에서 합니다.”

### 3.3 중간발표 정정: Supabase는 런타임 검색 DB가 아님
* **수정 전 오해**: 앱이 Supabase에 연결되어 실시간으로 사물을 검색하는 것처럼 발표되었습니다.
* **수정 후 표현**: 현재 앱은 Supabase·PostgREST·백엔드 서버 없이, 품목·조례·MOIS 일정·지역 문의처를 통합한 **로컬 SQLite DB**를 조회합니다.
* **Gemini의 역할**: Gemini API는 애매한 사진을 “종이컵”, “플라스틱병” 같은 한글 키워드로 바꾸는 보조 역할만 합니다. 사용자에게 보이는 안내문은 DB 문구를 기준으로 조합합니다.
* **향후 Supabase 사용 가능성**: 행안부 DB가 730여 개에서 1500개 수준으로 확장될 때, Supabase는 앱 자체 DB를 갱신하기 위한 임시 데이터 정제·업데이트 작업에만 활용할 수 있습니다. 앱이 실행 중 Supabase를 계속 조회하는 구조가 아닙니다.

## 4. 공공데이터 검색 및 API 연동 파이프라인

### 4.1 공공데이터 검색 및 수집 방식
* 행정안전부 공공데이터 포털 및 지자체 조례 사이트에서 제공하는 분리배출 기준표와 API 엔드포인트를 조사했습니다.
* `scripts/wasteguide_*_crawler.py` → `finalize_app_db.py` → `import_region_extras.py` 파이프라인으로 품목·조례·MOIS 일정을 정제해 `wasteguide.sqlite3`에 담습니다.

### 4.2 프로젝트에 적용된 4대 핵심 API
1. **Geocoder + SQLite**: GPS로 시·군·구를 판별한 뒤 번들 DB에서 조례·MOIS 일정을 조회합니다.
2. **Google ML Kit (`object-detection`) & AndroidX CameraX**: 안드로이드 폰의 카메라 라이브 프리뷰를 제어하고, 화면 속 사물의 바운딩 박스 좌표를 실시간으로 포착합니다.
3. **Gemini 3.1 Flash Lite API**: 로컬 DB로 판별이 애매한 복잡한 재질의 객체를 크롭하여 멀티모달 비전으로 전송, 빠르고 정확한 예외 판별을 수행합니다.
4. **E-순환거버넌스 연동 API**: AI가 '폐가전제품'으로 인식했을 때, 정부 운영 무상 방문 수거 기준표(단일 1개 가능, 소형 5개 이상 가능 팁)를 바텀시트에 자동으로 병합 호출합니다.

## 5. UI/UX 디자인 수정 및 디자인 시스템 조율

```
[ 디자인 토큰 (Tokens.kt) ]
  ├── 글래스모피즘 (Glassmorphism) : 반투명 블러 패널 + 슬릭한 다크 네이비 (Primary: #1E293B)
  ├── AI 자동 포착 : 녹색 네온 박스 (NeonGreen: #7CFF6B)
  └── 사용자 직접 드래그 : 주황색 네온 박스 (NeonOrange: #FF6D00) + "이 영역 분석" 팝업
```

### 5.1 UI 프레임워크 및 테마
* **100% Jetpack Compose**: 안드로이드의 최신 선언형 UI 프레임워크를 사용했습니다.
* **글래스모피즘 & 다크 테마**: 복잡하고 딱딱한 법률 앱 느낌을 없애고, 반투명 블러 효과와 고급스러운 다크 네이비 배경을 결합하여 세련된 무인 안내기 느낌을 연출했습니다.

### 5.2 인터랙티브 시각 피드백 (바운딩 박스 컬러링)
* **녹색 네온(`NeonGreen`)**: 카메라를 비췄을 때 AI가 자동으로 사물을 포착하고 트래킹하고 있음을 알리는 시그널입니다.
* **주황색 네온(`NeonOrange`)**: 여러 물체 중 사용자가 직접 화면을 슥 드래그하여 관심 영역을 지정했을 때 생성됩니다. AI의 선택과 사용자의 선택을 색상으로 명확히 분리하여 조작감과 UX 피드백을 극대화했습니다.

### 5.3 핀치 줌(Pinch Zoom) 및 오터치 방어 (UX Guardrails)
* **핀치 줌 계층 최적화**: 돋보기가 필요한 고령층을 위해 바텀시트를 두 손가락으로 확대할 수 있습니다. 이때 단순 시각 확대(`graphicsLayer`)만 쓰면 하단이 잘리는 버그가 발생하므로, `Modifier.layout`을 도입해 부모 뷰에 늘어난 실제 높이를 동적으로 보고하여 맨 밑 법적 고지까지 완벽하게 스크롤되도록 설계했습니다.
* **오터치 및 미세 드래그 방어**: 화면을 미세하게 잘못 건드려 32px 이하의 작은 박스가 생기면 화면이 먹통이 되는 문제를 막기 위해, 즉시 자동 취소(초기화)되거나 빈 배경 탭 시 해제되는 방어 로직을 추가했습니다.

## 7. 발표 추가 페이지: Eval 테스트 케이스 5개

| 평가 항목 | 테스트 기준 | 낮은 점수 원인 | 수정 내용 |
|---|---|---|---|
| 핀치 줌 | 확대/축소 후 하단 고지까지 스크롤되는가 | 확대 시 화면 하단이 잘리는 문제 | `layout` 기반 줌 컬럼으로 실제 높이를 반영하여 전체 스크롤 가능 |
| Supabase 배제 | 비행기 모드에서도 규칙 카드가 표시되는가 | 클라우드 검색처럼 설명되어 오해 발생 | Supabase 런타임 미사용, SQLite 번들 + Gemini 보조 구조로 문서 정정 |
| DB 통합 | 품목·지역·MOIS 일정·문의처가 한 카드에 결합되는가 | 지역별 안내가 분산되어 흐름이 복잡함 | 모든 app-ready 데이터를 `wasteguide.sqlite3`로 통합하고 지역코드로 매칭 |
| 반복 안내문 단순화 | 동일 주의문이 한 번만 표시되는가 | 세척·라벨 제거 등 공통 안내가 반복됨 | 공통 안내문은 1회 카드로 단순화 |
| 오터치 방어 | 작은 드래그·빈 화면 탭에서 정상 복구되는가 | 작은 박스 생성 후 UX가 막히는 문제 | 32px 이하 드래그 자동 취소, 빈 배경 탭 시 선택 해제 |

## 8. 발표 추가 페이지: 앱 실행 흐름

```
아이콘 클릭
  → CameraX 카메라와 GPS/Geocoder 준비
카메라에 사물 비추기
  → ML Kit이 후보 물체를 초록 박스로 추적
분석된 사물 클릭
  → 애매한 경우에만 Gemini API가 이미지를 한글 키워드로 변환

  → GPS 지역코드로 조례·MOIS 배출 일정·문의처 조회

  → 분류, 비우기/헹구기, 배출 요일, E-순환 전화 연결을 한 화면에 제공
```
