## 2. 현재 버전 상세 (v2.0 / v3)

### 2.1 시스템 구성

#### AI / Vision 및 매칭 엔진
- **엔진 구성**: **자체 구축 DB (730개 핵심 품목) + Gemini 1.5 Flash 멀티모달 API**
- **구조적 전환 (YOLO 제거)**: 
  - 기존의 YOLOv8n 가중치 파일 로딩 및 실시간 온디바이스 추론을 완전히 배제.
  - 카메라 라이브 프리뷰에서 사물이 포착되거나, 사용자가 커스텀 드래그로 영역을 지정하면 즉시 이미지 바이트와 좌표를 기반으로 자체 730개 품목 DB에서 1차 고속 키워드 매칭 수행.
  - 복잡한 재질이나 세부 판별이 필요한 경우 Gemini 1.5 Flash API로 경량화된 비전 요청 전송.

#### API 및 클라우드 서비스
- **행정안전부(행안부) API**: 
  - 기기에서 획득한 GPS 좌표를 기반으로 실시간 행정동/지자체 코드 및 최신 분리수거 규정 메타데이터 수집.
- **Supabase Cloud DB**:
  - 전국 지자체의 분리수거 룰셋(`ItemRule`), 자체 730개 품목 매핑 테이블, 사용자 피드백 및 정정 요청 로그를 저장·제공하는 핵심 BaaS 저장소.
- **E-순환거버넌스 연계 모듈**:
  - 대형폐기물 중 '폐가전제품' 분류 시 Supabase에서 E-순환거버넌스 무상 방문 수거 기준표(단일/다량 수거 팁)를 자동으로 호출하여 바텀시트에 병합 렌더링.

#### 런타임 및 클라이언트
- **언어**: Kotlin (Android Native), Python 3.11 (백엔드 데이터 수집 파이프라인)
- **프레임워크**: **Android Jetpack Compose (Native UI)**
- **UI 핵심 기술**: `Canvas` 기반 실시간 네온 바운딩 박스 드로잉, `Modifier.layout` 기반 핀치 줌 동적 높이 확장 스크롤.

#### 저장소 아키텍처
- **원격 DB (Supabase)**: `region_rules`, `items_730_master`, `user_clarify_logs` 테이블.
- **로컬 캐시 (SQLite / Room)**: 자체 730개 품목 사전 및 최근 조회 지자체 룰셋(`WasteGuideDb`).

### 2.4 주요 기술 결정과 근거 (Trade-offs)

**결정 1: YOLO 추론 완전 제거 및 자체 DB(730개) + Gemini 하이브리드 도입**
- **대안들**:
  - (A) YOLOv8 계속 유지: 실시간성은 좋으나, 730개 품목으로 확장 시 모바일 기기 발열 및 메모리 부족 현상 발생.
  - (B) Gemini API 단독: 모든 프레임을 서버로 전송하여 비용 폭증 및 네트워크 지연 발생.
  - (C) 자체 730개 DB 로컬 매칭 + Gemini 선택적 폴백: 초고속 1차 필터링과 강력한 AI 추론의 결합.
- **선택 근거**: 모바일 환경에서 배터리와 발열 관리는 필수적이며, 자체 730개 품목 DB를 구축함으로써 85% 이상의 일상 쓰레기를 0.1초 이내에 로컬에서 매칭할 수 있게 됨. 예외 상황에서만 Gemini를 호출하여 비용과 성능을 모두 잡음.

**결정 2: 행정안전부 API 및 Supabase 연동**
- **선택 근거**: 지자체별 분리수거 규정은 자주 변경되므로 앱 내부에 하드코딩하는 것은 불가능함. 행안부 API로 정확한 행정동을 파악하고, 클라우드 BaaS인 Supabase에서 즉시 최신 룰을 가져옴으로써 관리 비용과 데이터 정합성을 획기적으로 개선함.

**결정 3: 핀치 줌 구조에서 `Modifier.layout` 도입**
- **선택 근거**: `graphicsLayer`만으로 화면을 확대하면 부모 스크롤 뷰가 콘텐츠의 늘어난 높이를 알지 못해 하단이 잘리는 치명적인 Compose UX 버그가 있었음. `Modifier.layout`을 통해 `scale`에 비례하여 실제 측정 크기를 부모에게 보고하도록 설계하여 완벽한 스크롤 경험을 달성함.

**결정 4: 커스텀 드래그 영역에 주황색 네온(`NeonOrange`) 적용**
- **선택 근거**: 사용자가 화면을 터치/드래그했을 때 AI 자동 인식(녹색)과 동일한 색상이 나오면 자신의 조작이 반영되었는지 인지하기 어려움. 명도 대비가 우수하고 조작감을 강조하는 주황색 네온을 채택하여 UX 피드백을 강화함.

## 3. 가드레일 (Guardrails)

### 3.1 Pre-AI (입력 단계 필터링)
- **PII 및 프라이버시 보호**: 카메라 프레임이나 캡처 이미지 내의 신분증, 택배 송장, 처방전 봉투 등 개인정보가 감지될 경우 클라우드 전송 전 온디바이스에서 자동 블러/차단 처리.
- **위험물 즉시 안내**: 가스통, 배터리, 주사기 등 위험 폐기물 식별 시 자체 DB 필터를 통해 즉시 특수 폐기 안내 스크린 노출.

### 3.2 Post-AI (출력 단계 검증)
- **Supabase DB 교차 검증**: Gemini API가 환각(Hallucination)으로 잘못된 품목명을 반환하더라도, Supabase의 `items_730_master` 테이블에 존재하지 않는 품목은 바텀시트에 노출하지 않고 폴백 안내로 전환.
- **E-순환거버넌스 정합성 체크**: 폐가전제품 안내 시 공식 무상 수거 기준표 데이터와 일치하는지 스키마 검증 후 렌더링.

## 4. 하네스 및 평가 (Evals)

### 4.1 자체 DB 730개 벤치마크 현황
- 총 테스트 품목: **730개 핵심 품목** × 다양한 조명/배경 환경 이미지 3장 = **2,190개 테스트셋**.
- **평가 지표**:
  - **pass@1 (1차 매칭 정확도)**: 자체 DB + Gemini 결합 기준 **94.2%** 달성 (목표 90% 초과).
  - **Latency (평균 응답속도)**: 로컬 DB 매칭 45ms, Gemini 폴백 포함 평균 780ms.
  - **지역 매핑 정합성**: 행안부 API ↔ Supabase 룰 매핑 일치율 **99.8%**.
